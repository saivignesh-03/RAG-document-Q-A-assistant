What it does

Three ideas make this more than a wrapper around an LLM prompt.

01

Hybrid retrieval

Semantic search finds meaning, BM25 finds exact terms, and Reciprocal Rank Fusion combines both rankings — so a rare name and a vague concept are both findable.

02

Reranked for accuracy

A cross-encoder reads the question and each candidate chunk together, catching relevance that comparing rank positions alone can't.

03

Grounded, not guessing

Answers come only from retrieved context. If the document doesn't contain it, the assistant says so — instead of filling the gap from its own training.

The signal chain

Every question travels through the same eight stages before an answer comes back.

01
PDF → CHUNKS
Document loaded and split into overlapping ~800-character chunks.
02
EMBEDDINGS
Each chunk converted into a 768-dimension vector, stored in ChromaDB.
03
SEMANTIC SEARCH
Top 10 chunks by cosine similarity to the question's vector.
04
KEYWORD SEARCH
Top 10 chunks by BM25, for exact terms embeddings can miss.
05
RRF FUSION
Both rankings merged by rank position — a shortlist of 5, no score-scale tuning needed.
06
RERANK
A cross-encoder re-scores the shortlist by reading question and chunk jointly — top 3.
07
GENERATION
Llama 3.1 answers using only the retrieved context, or declines if it's not enough.
08
DELIVERY
Served through a FastAPI backend and a Streamlit chat interface.

From the build log

A few real entries from actually building this — not a polished summary.

[BUG]keyword search always returned k results, even chunks with a score of exactly zero.

Fixed by filtering out any chunk that didn't genuinely score above zero before ranking.

[LIMITATION]two genuinely relevant chunks tied under RRF fusion.

Not a bug — RRF only sees rank position, never the actual text. Solved by adding the cross-encoder reranker.

[INCIDENT]uvicorn --reload spawned two processes that both tried to load the LLM onto the GPU at once.

Crashed Ollama with a CUDA initialization error. Fixed by running without --reload.
